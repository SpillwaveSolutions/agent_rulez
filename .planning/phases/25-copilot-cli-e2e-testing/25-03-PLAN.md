Modify copilot_adapter_check() in e2e/lib/copilot_adapter.sh to verify authentication state after the PATH check. The fix uses a two-stage approach:

Stage 1 (preferred): If `gh` is in PATH, run `gh auth status 2>/dev/null`. A zero exit code means GitHub OAuth is active, which is sufficient for Copilot CLI. A non-zero exit means unauthenticated — return 1 with a clear message.

Stage 2 (fallback): If `gh` is NOT in PATH, perform a short timeout probe: `timeout 5 copilot -p "ping" --allow-all-tools 2>/dev/null`. If it exits 1 immediately (within the 5s timeout), treat as unauthenticated and return 1. If it times out (exit 124) or exits 0, treat as authenticated and return 0.

The final function should:
1. Check `command -v copilot` — return 1 with install instructions if missing (existing behavior, keep as-is)
2. Print copilot version (existing behavior, keep as-is)
3. If `command -v gh > /dev/null 2>&1`: run `gh auth status 2>/dev/null`; if non-zero, print error "GitHub not authenticated (copilot requires GitHub OAuth). Run: copilot auth login" and return 1
4. Else (no gh): run `timeout 5 copilot -p "ping" --allow-all-tools > /dev/null 2>&1`; capture exit code; if exit code is 1, print error "Copilot CLI unauthenticated (probe exited 1). Run: copilot auth login" and return 1; if exit code is 0 or 124 (timeout = likely authenticated, just slow), continue
5. Print "copilot_adapter: authentication check passed" and return 0

Also update the function header comment (lines 19-24) to reflect that it now checks both PATH presence and authentication state.

Do NOT modify require_copilot_cli, setup_copilot_hooks, or invoke_copilot_headless.

Reference: gemini_adapter_check in e2e/lib/gemini_adapter.sh checks GEMINI_API_KEY after the PATH check — this auth check is the analogous mechanism for Copilot.
